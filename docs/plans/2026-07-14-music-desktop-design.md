# 音乐搜索与下载桌面应用设计

日期：2026-07-14
状态：需求、总体架构与技术栈已确认，应用名称和视觉设计待后续确定

## 1. 项目目标

使用完整的 Tauri 桌面方案开发一款面向 Windows 和 macOS 的音乐搜索与下载应用。应用调用已获授权的 GD Studio 音乐接口，核心能力仅包括搜索、选择和顺序下载，不扩展播放器、账号体系或媒体库管理功能。

发布侧使用 GitHub Actions 和 GitHub Releases，为用户提供标准安装包、版本发布和应用内自动更新。

## 2. 范围

### 2.1 包含

- 启动初始化页面和主页面。
- 按关键词搜索单曲、专辑或歌单。
- 每次选择一个音源，默认 QQ 音乐。
- 可选搜索数量和音质。
- 单选、多选、全选搜索结果并批量下载。
- 下载严格顺序执行，不并发下载歌曲。
- 下载进度、统计、取消、失败重试和运行日志。
- 音频封面嵌入和同名歌词文件。
- 基于目标目录现有文件的实时判重。
- Windows x64 和 macOS Universal 安装包。
- GitHub Release 自动发布及 Tauri 自动更新。

### 2.2 不包含

- 音乐播放和播放列表管理。
- 用户登录、云同步、下载历史或本地媒体库。
- 应用设置持久化和本地用户配置文件。
- Windows ARM64 首发安装包。
- 首版 Windows 正式代码签名和 macOS Developer ID 公证。

## 3. 已核验的接口方案

接口入口为：

```text
https://music-api.gdstudio.xyz/api.php
```

真实页面为：

```text
https://music.gdstudio.xyz/
```

不能只依赖接口文档判断音源或参数能力。真实页面会加载签名逻辑，并通过页面内的 `crc32(input)` 生成动态参数 `s`。部分音源即使省略 `s` 也可能暂时成功，但应用必须统一携带 `s`，与真实网页的请求行为保持一致。

采用以下调用方式：

1. 启动隐藏 WebView 加载官方页面。
2. 从真实页面发现当前音源，并验证签名函数可调用。
3. 通过页面提供的 `crc32()` 计算签名，不在 Rust 中复制或长期维护签名算法。
4. Rust 后端执行带签名的表单 POST 请求、响应解析和文件下载。
5. 所有音源请求统一包含 `s`。

隐藏 WebView 和签名桥接被限制在 `music` 模块内部。页面结构或签名函数变化时，错误应收敛为明确的初始化或接口错误，不影响其他模块边界。

搜索类型在界面中固定展示“单曲、专辑、歌单”，不根据音源动态禁用。如果真实接口拒绝某个音源与类型的组合，界面直接展示接口错误。

接口存在请求额度限制，当前按全局每 5 分钟最多 50 次请求设计。搜索、获取音频地址、封面和歌词均纳入同一个限流器。达到额度后，任务进入等待/冷却状态，而不是无控制地重试。

## 4. 页面与交互

### 4.1 启动页

启动页负责：

- 初始化系统日志。
- 初始化隐藏签名 WebView。
- 发现并加载真实音源。
- 初始化下载与更新服务。
- 后台检查更新。

签名或音源初始化失败时，启动页提供“重试”和“退出”。更新检查失败不阻止进入主页面，只记录警告日志。

### 4.2 主页面搜索区

每次启动使用固定默认值，不读取上次运行状态：

| 配置 | 默认值 | 规则 |
| --- | --- | --- |
| 关键词 | 空 | 搜索前必须有有效内容 |
| 音源 | QQ 音乐 | 每次只能选择一个音源 |
| 类型 | 单曲 | 固定选项：单曲、专辑、歌单 |
| 搜索数量 | 20 | 仅要求正整数，不设置应用侧上限 |
| 音质 | 320 | 可选 128、192、320、740、999 |
| 下载封面 | 勾选 | 封面嵌入音频文件 |
| 下载歌词 | 勾选 | 生成同名 `.lrc` 文件 |
| 基础下载目录 | 系统音乐目录 | 运行期间允许修改 |

所有配置仅存在于当前运行内存中。重启应用恢复默认值，不生成用户配置文件。

正在下载时锁定搜索条件和新搜索操作。用户如需更换关键词，必须先取消当前队列。

### 4.3 搜索结果

结果列表固定展示：

- 选择框。
- 序号。
- 名称/歌手。
- 专辑。
- 来源/状态。
- 时长。

歌曲 ID 不单独显示，但保留在内部模型中，供文件命名、判重、下载和日志定位使用。时长显示为 `mm:ss`，缺失时显示 `--:--`。

列表支持单选、多选和全选，并同时显示结果总数和已选数量。

### 4.4 下载区

- 用户点击下载后，把选中歌曲一次性加入队列。
- 队列严格一次只处理一首歌。
- 支持取消当前歌曲、取消全部、单条失败重试和全部失败重试。
- 重新搜索或重启应用会重置下载统计。
- 下载状态包括等待、限流冷却、下载中、成功、失败、跳过和已取消。

进度区域展示：

- 总任务数。
- 等待数。
- 成功数。
- 失败数。
- 跳过数。
- 当前歌曲字节进度、百分比和速度。

音频文件成功落盘即计为成功。封面或歌词失败只显示警告并写入日志，不把歌曲改为失败状态。

## 5. 下载目录、命名与判重

### 5.1 默认路径

- Windows：系统 Music 目录。
- macOS：`~/Music`。

实际歌曲保存到：

```text
<用户选择的基础目录>/<清理后的搜索关键词>/
```

中间不增加应用名目录。关键词、歌曲名称和 ID 必须经过跨平台路径清理，防止非法字符、保留名称和路径穿越。

### 5.2 文件命名

音频文件名：

```text
【歌曲ID】歌曲名称.<实际扩展名>
```

扩展名必须根据真实媒体响应或内容识别，不能假定全部是 MP3。接口可能返回 MP3、FLAC、M4A/MP4 等格式。

歌词文件使用音频基础名称：

```text
【歌曲ID】歌曲名称.lrc
```

### 5.3 判重

判重键为：

```text
歌曲 ID + 清理后的歌曲名称 + 实际音频扩展名
```

规则：

- 只有最终音频文件参与判重。
- `.part` 临时文件和 `.lrc` 歌词文件不参与判重。
- 同 ID、同名称但扩展名不同不跳过，例如已有 MP3 时仍可下载 FLAC。
- 命中完全一致的最终文件时跳过，不覆盖或修改已有文件。
- 每次搜索时扫描目标关键词目录并构建内存集合。
- 成功下载后把新文件加入集合。
- 重新搜索、切换关键词或重启应用时释放并重新构建集合。

### 5.4 安全写入

音频先流式写入同目录 `.part` 文件。下载和必要校验完成后，再原子重命名为最终文件。取消或失败时保留还是删除临时文件由实现阶段根据可恢复性确定，但临时文件永不计入成功和判重。

## 6. 封面、歌词与媒体元数据

- 封面不单独保存图片文件，直接嵌入音频元数据。
- MP3 使用 ID3 图片帧；FLAC 使用 PICTURE 元数据块；其他格式按实际容器能力处理。
- 歌词采用同名 `.lrc` 独立文件，优先保留接口返回的时间轴内容。
- 如果某种真实媒体格式暂不支持封面写入，歌曲仍算成功，并记录可定位的警告。
- 元数据写入不得破坏已经完成下载的音频；必要时使用临时文件完成修改后再替换。

## 7. 日志

日志需要落盘并在主页面提供实时查看，日志属于诊断数据，不属于用户配置。

日志目录：

- Windows：`%LOCALAPPDATA%/<应用名>/log/`。
- macOS：`~/Library/Logs/<应用名>/`。

开发运行、手动构建和安装后的正式程序均使用同一平台规则，不写到可执行文件同级目录。日志采用滚动文件，避免无限增长。

界面提供：

- 查看当前运行日志。
- 复制日志内容。
- 打开系统日志目录。

日志至少记录初始化、音源、请求阶段、限流、歌曲 ID、下载阶段、媒体处理、更新检查和错误链。避免记录更新私钥或其他敏感凭据。

## 8. 总体模块架构

### 8.1 Rust 后端

后端保留六个顶级业务模块，具体职责使用模块内部文件组织，避免过度拆分：

1. `app`：启动、服务装配、运行状态、命令和事件协调。
2. `music`：隐藏 WebView 签名、音源发现、API 客户端和全局限流。
3. `download`：顺序队列、判重、流式下载、取消、重试和统计。
4. `media`：真实格式识别、封面嵌入和歌词生成。
5. `system`：平台目录、文件名清理和日志。
6. `update`：版本检查、更新下载、安装和重启协调。

下载队列和进度由 Rust 后端维护为唯一真实状态。前端发送命令并订阅状态事件，组件重绘或页面状态变化不得打乱队列。

### 8.2 前端

前端采用“页面组合 + 功能划分”，具体目录结构在选定前端技术栈后确定。

页面层：

- 启动页。
- 主页面。

功能边界：

- 搜索表单。
- 结果与选择。
- 下载和进度。
- 日志查看。
- 更新提示。
- Tauri 通信桥接。
- 真正复用的通用组件和工具。

原则：

- 页面只负责组合与布局，不直接实现 API 或文件操作。
- Tauri 命令和事件集中封装，不散落在组件中。
- 下载状态以后端为准。
- 搜索表单、结果选择、弹窗等临时界面状态由前端管理。
- 不为了形式提前拆分小文件，也不把整个主页面写成一个大组件。
- 只有两个页面，不引入前端路由，直接根据初始化状态切换页面。

## 9. 关键数据和状态

### 9.1 搜索结果

核心字段包括：歌曲 ID、名称、歌手列表、专辑、音源、时长、可用状态以及接口下载所需的原始标识。

### 9.2 下载任务

任务保存搜索快照和歌曲快照，至少包括音源、音质、目标目录、是否下载封面/歌词、任务状态、尝试次数、错误信息和字节进度。用户在任务开始后修改搜索表单，不得改变已经入队的任务。

### 9.3 事件

后端以事件通知前端：初始化阶段变化、日志追加、队列变化、任务状态变化、当前字节进度、统计变化、限流冷却、更新检查和更新下载进度。

事件载荷使用稳定的数据结构，不把 Rust 内部错误类型或第三方接口原始响应直接暴露给前端。

## 10. 错误边界

- 初始化错误：停留启动页，可重试或退出。
- 更新初始化/检查错误：记录警告，不阻止进入应用。
- 搜索错误：保留搜索条件，展示可理解的接口错误。
- 单曲下载错误：标记当前任务失败，继续处理后续任务。
- 封面/歌词错误：歌曲成功，附件警告。
- 文件系统错误：显示目标路径和可操作原因，但不暴露无关内部细节。
- 限流：显示等待时间，不作为歌曲失败。
- 页面签名逻辑变化：明确标记为签名初始化或签名请求失败，便于日志定位。

## 11. 版本与自动更新

`src-tauri/tauri.conf.json` 中的 `version` 是唯一应用版本源，使用语义化版本：

- `0.1.0`：首个测试版本。
- `0.1.1`：兼容性修复。
- `0.2.0`：新增功能。
- `1.0.0`：首个稳定版本。

发布步骤：

1. 手动修改应用版本并提交。
2. 创建同版本 Git 标签，例如 `v0.1.1`。
3. GitHub Actions 验证标签与配置版本一致。
4. 从同一提交构建全部平台。
5. 生成安装包、更新包和更新签名。
6. 创建 GitHub Release，并生成 `latest.json`。
7. 客户端从 GitHub Release 的 `latest.json` 检查更高版本。

默认不允许自动降级。若 `1.2.0` 出现问题，应基于修正代码发布 `1.2.1`，而不是让客户端退回 `1.1.0`。

更新体验：

1. 启动后后台检查更新。
2. 发现新版本后后台下载，不阻塞主功能。
3. 下载完成提示“立即重启”或“稍后”。
4. 选择稍后时，可在退出时安装，或下次启动再次提示。
5. 更新失败只记录错误并提示，不影响当前版本继续使用。

Tauri 更新签名与 Windows/macOS 系统代码签名相互独立。更新签名从第一版启用且不可关闭：公钥嵌入应用，私钥和密码仅保存在 GitHub Actions Secrets 中。

## 12. CI/CD 与安装包

### 12.1 目标产物

- Windows x64：标准 NSIS `.exe` 安装程序。
- macOS Universal：单个 `.dmg`，同时支持 Intel 和 Apple Silicon。
- 对应平台的 Tauri 更新包、`.sig` 和 `latest.json`。

### 12.2 流水线

- 普通提交/拉取请求：安装依赖、静态检查、单元测试和构建验证。
- `v*` 标签：执行版本一致性验证和跨平台发布构建。
- 任一必需平台失败时，不发布不完整的正式版本。
- Release 资产来自同一 Git 提交。
- 发布后的同版本资产不可覆盖，修复必须提升版本。

### 12.3 首版系统签名策略

当前发布者是中国大陆个人开发者。

- Windows：首版不做 Authenticode 代码签名，用户可能遇到 SmartScreen 和“未知发布者”提示。CI 中保留未来接入第三方公开信任代码签名服务的步骤和 Secrets 名称，但未配置时跳过。
- Azure Artifact Signing Public Trust 当前不支持中国大陆个人身份，因此不作为当前方案。
- macOS：首版使用免费的 ad-hoc signing。用户仍可能需要在“隐私与安全性”中手动允许应用。
- macOS 正式公开推广前，可加入个人 Apple Developer Program，再接入 Developer ID 签名和 notarization。CI 为相关证书和 Apple 凭据预留条件步骤。

系统代码签名缺失不影响 Tauri 更新包的密码学签名，但不能消除 SmartScreen 或 Gatekeeper 的系统信任提示。

## 13. 验证策略

实现阶段至少覆盖：

- 签名输入和真实请求构造测试。
- 多音源、三种搜索类型和不同数量边界测试。
- 文件名清理、路径穿越和 Windows/macOS 保留名称测试。
- 不同扩展名判重测试。
- `.part` 到最终文件的成功、取消和失败测试。
- 顺序队列保证任意时刻最多一首歌曲下载。
- 限流和冷却状态测试。
- 封面/歌词失败不改变音频成功状态。
- 搜索、重新搜索、重启后的状态重置。
- Tauri 命令与事件载荷测试。
- Git 标签和应用版本一致性测试。
- 更新签名错误、无更新、有更新和更新失败测试。
- Windows x64 安装验证和 macOS Intel/Apple Silicon 安装验证。

## 14. 技术栈

### 14.1 工具链

- Node.js 24 LTS。
- pnpm，并提交 `pnpm-lock.yaml`。
- Rust 1.97.0，通过 `rust-toolchain.toml` 固定版本。
- Tauri 2，并提交 `Cargo.lock`。
- CI 与开发环境使用相同的 Node 主版本、pnpm 版本和 Rust 精确版本。

当前开发机已经安装 Node.js 和 pnpm，但尚未安装 Rust/Cargo。开始搭建项目前需通过 rustup 安装 Rust，并安装 Windows 所需的 Microsoft C++ Build Tools；WebView2 使用系统运行时。

### 14.2 前端

- Svelte 5 + TypeScript 严格模式。
- Vite 构建静态 SPA，不使用 SvelteKit 或 SSR。
- Skeleton 组件库和设计系统。
- Tailwind CSS 4，并以 Skeleton 主题变量作为视觉令牌基础。
- Lucide Svelte 图标。
- 使用 `$state`、`$derived` 和 `.svelte.ts` 状态模块，不增加第三方状态库。
- 使用 `@tauri-apps/api`，所有命令和事件通过 `bridge` 层封装。
- Rust DTO 使用 `ts-rs` 生成 TypeScript 类型；Tauri 命令包装保持少量手写代码。
- Vitest + Svelte Testing Library 负责状态和组件测试。
- Playwright 在浏览器模式下通过模拟 Tauri Bridge 验证页面流程。
- `svelte-check`、ESLint 和 Prettier 作为日常与 CI 质量检查。
- 首版界面使用简体中文，用户可见文案集中管理，不在各组件内重复硬编码。

前端不引入额外表单校验库。搜索参数使用类型明确的小型校验函数。结果列表采用 Skeleton 样式加语义化表格，不引入复杂数据表格依赖。

### 14.3 Rust 后端

- Tokio：异步任务、通道和单消费者下载工作器。
- `tokio-util::CancellationToken`：取消当前任务和整个队列。
- Reqwest + rustls：表单 POST、JSON、封面、歌词和流式音频响应。
- Serde + serde_json：外部接口和 IPC 序列化。
- thiserror：结构化领域错误和可序列化前端错误映射。
- ts-rs：从 Rust DTO 生成 TypeScript 类型。
- Lofty：MP3、FLAC、MP4/M4A、Ogg 等媒体标签和封面写入。
- Wiremock：音乐 API 和下载响应的黑盒 HTTP 测试。
- `tauri-plugin-log` + `log`：Rust/前端统一日志、自定义目录和大小滚动。
- `tauri-plugin-dialog`：原生目录选择。
- `tauri-plugin-opener`：打开下载目录和日志目录。
- `tauri-plugin-updater` + `tauri-plugin-process`：更新、安装和重启。
- `tauri-plugin-single-instance`：避免多个应用进程同时写入相同目标文件。

以下能力作为项目业务规则自行实现并测试，不增加通用依赖：

- 每 5 分钟最多 50 次请求的滑动窗口限流。
- 单消费者顺序下载队列。
- 跨平台文件名清理和 Windows 保留名称处理。
- 常见音频容器文件头识别和实际扩展名决策。
- ID、名称和扩展名判重。
- `.part` 写入、取消清理和原子重命名。
- 下载任务状态机和统计。

### 14.4 明确不采用

- 不使用数据库、SQLite、Tauri Store 或任何配置持久化。
- 不使用 `tauri-plugin-http`，音乐网络请求全部在 Rust 中完成。
- 不向前端开放 `tauri-plugin-fs` 通用文件权限。
- 不开放 Shell 插件，只提供明确的目录打开能力。
- 不使用 FFmpeg，不转换接口返回的音频格式。
- 不使用仍处于 RC 阶段的 tauri-specta；使用稳定的 ts-rs 生成共享 DTO。

Tauri capability 只允许主窗口调用实际需要的命令和官方插件。加载真实音乐页面的隐藏签名 WebView 使用单独 capability，只开放带关联 ID 的签名/音源结果通道，不拥有文件、日志、更新或系统操作权限。
