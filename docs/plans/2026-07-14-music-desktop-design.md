## 1. 产品目标与身份

音觅是一款由 GD 音乐台接口驱动的 Windows 与 macOS 桌面音乐搜索、批量下载工具。首版聚焦“搜索—选择—下载—查看结果”这一条完整链路，不承担播放器、账号系统或本地音乐库职责。GD Studio 是音乐接口提供方，不在本文中据此推定应用发行主体。

项目通过 GitHub Actions 构建，通过 GitHub Releases 发布安装包和应用内更新。

| 日志目录名称 | 音觅 |
| 工程名 | `yinmi` |

## 2. 范围与非目标

### 2.1 首版范围

- 启动初始化页与主界面。
- 关键词搜索，支持单曲匹配、专辑匹配、歌单匹配三种模式。
- 从固定音源列表中选择一个音源，默认使用网易云音乐。
- 配置搜索数量、下载音质、封面写入、歌词下载和基础下载目录。
- 搜索结果选择、全选与批量加入下载。
- 单任务顺序下载、进度展示、取消、失败重试、统计和日志查看。
- 将封面嵌入音频文件，将歌词保存为同名 `.lrc` 文件。
- 目标文件去重、安全临时文件落盘及原子提交。
- Windows x64 和 macOS Universal 构建、GitHub Release 发布和应用内更新。

### 2.2 首版非目标

- 音乐播放、播放列表管理。
- 用户登录、云同步、搜索历史或本地音乐库。
- 配置持久化；每次启动均使用默认配置。
- Windows ARM64。
- 首版 Windows Authenticode 代码签名。
- 首版 Apple Developer ID 签名与公证。

## 3. 支持平台、工具链与技术方案

### 3.1 支持平台

| 平台 | 首版支持范围 |
| --- | --- |
| Windows | Windows 10 22H2、Windows 11，x64 |
| Windows WebView | WebView2 `111.0.1661.0` 或更高版本 |
| macOS | macOS 13.3 或更高版本，Intel 与 Apple Silicon |
| macOS 产物 | 单一 Universal 应用与对应 Universal 更新包 |

Windows 安装包使用 WebView2 在线引导程序，不内置完整 Evergreen Runtime。安装器或引导程序负责处理运行时缺失；应用能启动但检测到版本不足时，由启动页给出明确升级指引，不得只显示空白窗口。

前端构建目标为 Chrome 111 与 Safari 16.4，以匹配首版 WebView 与 Tailwind CSS 4 的运行基线。

### 3.2 固定工具链

- Node.js 24 LTS，使用 `pnpm` 并提交锁文件。
- Rust 1.97，通过 `rust-toolchain.toml` 固定。
- Tauri 2，提交 `Cargo.lock`。
- 本地开发、持续集成和发布工作流使用相同的主版本基线。

工具链补丁版本可通过常规依赖维护升级，但升级后必须重新通过第 11 章中的验证。

### 3.3 前端方案

Tauri 2、Svelte 5、TypeScript 严格模式和 Vite SPA 是首版固定架构。其余库是首选实现；如果可行性验证证明存在平台兼容、维护或安全问题，可以通过简短设计决策记录替换，但不得改变本文的用户行为和验收结果。

- Svelte 5 与 TypeScript 严格模式。
- Vite 单页应用。
- Skeleton UI 与 Tailwind CSS 4。
- Lucide 图标。
- 使用 Svelte 状态管理界面瞬态状态。
- 通过 `@tauri-apps/api` 调用受控的 Rust 命令。
- 使用 `ts-rs` 从 Rust 领域类型生成前端类型。
- Vitest 与 Testing Library 负责前端单元和组件测试。
- 所有用户可见文案集中维护为简体中文资源。

首版只有启动页和主界面，不要求引入路由；未来增加独立页面时可按需要引入。

### 3.4 Rust 后端方案

Rust、Tauri 2、后端统一持有权威状态，以及所有第三方网络请求由 Rust 侧通过安全 TLS 发起，是首版固定架构。以下具体库是首选实现，允许按第 3.3 节相同规则替换。

- Tokio 异步运行时。
- `tokio-util::sync::CancellationToken` 管理可取消任务。
- Reqwest 与 rustls 处理网络请求。
- Serde 处理序列化与响应解析。
- Thiserror 定义稳定错误类型。
- `ts-rs` 导出跨层类型。
- Lofty 写入音频元数据与封面。
- `log` 与 Tauri log 插件记录滚动日志。
- Tauri dialog、opener、updater、process、single-instance 等官方插件提供系统能力。
- Wiremock 提供 Rust 网络契约测试。

### 3.5 架构边界

- 首版无业务数据库和配置持久化，因为产品范围不需要持久化状态。
- 所有音乐接口、媒体下载与更新网络请求由 Rust 发起；前端不得直接请求第三方音乐接口。
- 前端不获得通用文件系统、Shell 或任意网络访问能力。
- 首版不转码，不依赖 FFmpeg；文件扩展名必须反映实际下载到的音频格式。

## 4. 总体架构与信任边界

### 4.1 责任划分

后端按责任划分为以下能力域，实施时可以进一步合并或拆分：

- 应用编排：初始化、状态快照、命令入口和生命周期协调。
- 音乐集成：签名、GD 音乐台协议、音源注册、响应归一化。
- 下载引擎：不可变任务、单消费者队列、取消、进度和统计。
- 媒体与存储：路径处理、格式识别、元数据写入、去重和原子提交。
- 系统能力：日志、目录打开、单实例和平台差异。
- 更新能力：检查、下载、退出屏障和安装。
- 前端界面：启动状态、搜索配置、结果表格、下载栏和日志抽屉。

Rust 是第三方协议、文件状态、下载队列与更新流程的权威状态源。前端只持有表单输入、当前选择、界面展开状态等瞬态状态；从后端同步的领域状态不得由前端自行推演为事实。

### 4.2 信任边界

- 主窗口只获得其业务必需的 Tauri 权限。
- 用于执行官方签名脚本的隐藏 WebView 被视为不可信远程内容，必须是零能力上下文：无 Tauri IPC、无插件调用、无应用事件和无文件系统能力。
- 第三方音乐接口、媒体地址和响应内容均为不可信输入，必须经过协议校验、资源边界和安全错误转换。
- GitHub Release 更新清单与更新包必须通过 Tauri 更新签名验证。

### 4.3 关键数据流

1. 启动时初始化日志、签名运行环境、音源注册、下载引擎，并在后台检查更新。
2. 搜索时前端提交带唯一身份的搜索请求；Rust 完成签名、接口调用和归一化，返回 `SearchSnapshot` 与歌曲结果。
3. 用户加入下载时，Rust 从当前搜索快照、歌曲快照和下载选项生成不可变任务；单消费者依次处理临时文件、媒体元数据和最终提交，并发布状态。
4. 更新检查和下载在后台进行；请求安装或关闭应用时，统一退出屏障协调未结束的音乐任务与仍在进行的更新下载，进入安全状态后再退出并安装。

## 5. GD 音乐台协议与安全要求

### 5.1 协议基线

- API：`https://music.gdstudio.xyz/api.php`
- 官方页面：`https://music.gdstudio.xyz/`
- 当前协议基线：页面版本 `2026.06.16`，资源查询版本参数 `v=20260616`

以上内容于 2026-07-14 按官方页面实际加载的 `api.php` 与脚本复核。该协议是外部依赖，可能随官方页面变化。协议发生变化时，应先更新契约样例和测试，再更新实现；不得通过放宽解析或安全校验来掩盖不兼容。

### 5.2 操作契约

所有操作均为表单请求，并携带由官方页面脚本生成的签名字段 `s`。

| 操作 | 主要字段 | 结果语义 |
| --- | --- | --- |
| `search` | `types`, `name`, `source`, `count`, `pages`, `s` | 返回单曲记录 |
| `search_album` | `types`, `name`, `source`, `count`, `pages`, `s` | 返回匹配专辑中的单曲记录 |
| `search_playlist` | `types`, `name`, `source`, `count`, `pages`, `s` | 返回匹配歌单中的单曲记录 |
| `url` | `types`, `id`, `source`, `br`, `s` | 返回音频地址及可能的码率、大小信息 |
| `pic` | `types`, `id`, `source`, `size`, `s` | 返回封面地址 |
| `lyric` | `types`, `id`, `source`, `s` | 返回原文歌词及可能的翻译歌词 |

- `pages` 从 1 开始。
- 封面 `size` 首版使用 300。
- 音质码率值为 `128`、`192`、`320`、`740`、`999`。
- 搜索数量是期望返回的歌曲总数，不要求单次上游请求完成；后端可以分页获取并在达到期望数量、无更多结果或安全边界时停止。
- 三种搜索操作都必须归一化为歌曲列表，不向界面暴露专辑或歌单容器对象。

Rust 命令边界必须再次验证所有输入，不能只依赖前端控件：

- 关键词去除首尾空白后非空，长度不超过附录 B 的上限。
- 音源必须存在于固定注册表中。
- 搜索模式只能是三种固定枚举值。
- 搜索数量只能是 1–1000 的整数。
- 码率只能是上述五个固定值。
- 路径必须来自受控目录选择结果或通过同等路径校验。

分页按以下用户可见语义聚合：

- 期望数量按去重后的有效歌曲计数；同一搜索内相同音源与归一化歌曲 ID 重复出现时，只保留第一条有效记录。
- 原始空页、明确“无更多结果”、某页没有增加任何新歌曲、达到期望数量或达到安全页数上限时停止。
- 在尚无有效结果时发生分页请求失败，整次搜索失败；已有有效结果后发生失败，则返回已有结果并显示“结果不完整”警告。
- 上游正常耗尽时，少于期望数量仍是正常成功；因安全上限停止时，必须显示“结果不完整”警告。
- 安全页数上限由第 12 章第一阶段根据真实接口验证后固定，未完成该门控前不得实现正式分页链路。

完整字段编码、音源映射和请求正文样例见附录 A。

### 5.3 音源注册与搜索模式

音源由应用内固定注册表提供，包含稳定内部代码、中文显示名称和上游 `source` 值。界面不从网络动态发现或加载音源插件。单个音源不可用时，应表现为普通搜索或下载错误，不影响其他音源。

首版默认音源为网易云音乐。完整音源表见附录 A.1。

三种搜索模式的固定用户语义如下：

| 模式 | 用户语义 |
| --- | --- |
| 单曲匹配 | 按歌曲关键词查找单曲 |
| 专辑匹配 | 查找匹配专辑，并返回其中可下载的歌曲 |
| 歌单匹配 | 查找匹配歌单，并返回其中可下载的歌曲 |

### 5.4 响应归一化与错误判定

搜索歌曲记录按以下规则归一化：

- `id`：接受字符串或安全整数，统一转换为字符串。
- `name`、`source`：必须为字符串。
- `album`：接受字符串或 `null`。
- `artist`：接受字符串或字符串数组，统一转换为显示字符串和结构化艺人列表。
- `url_id`、`pic_id`、`lyric_id`：可选；缺失时对应能力不可用。
- `extra_data.duration`：接受整数或数字字符串，上游单位为秒；统一转换为内部毫秒整数。缺失时为未知，无法安全转换时视为字段无效。
- `extra_data.has_hires`：统一为布尔值，缺失时为 `false`。

搜索顶层结构不符合协议时，整个请求以“响应格式不兼容”失败。列表中同时包含有效和无效记录时，跳过无效记录并返回汇总警告；非空原始结果最终没有任何有效歌曲时，同样按“响应格式不兼容”处理，不显示为正常空结果。

其他响应规则：

- `url` 响应必须提供有效 URL；`br`、`size` 和 `source` 可以缺失。
- URL 为空表示所选音质不可用。
- 上游明确返回低于请求值的 `br` 时，不得静默降级；应提示所选音质不可用。上游提供 URL 但未提供 `br` 时允许继续，并以实际内容识别结果为准。
- `pic` 响应必须提供有效图片 URL。
- `lyric` 响应中的 `lyric` 与 `tlyric` 均可缺失；首版 `.lrc` 只保存非空原文 `lyric`，不合并翻译歌词。原文为空时不创建空文件，并记录附件不可用警告。
- 非 2xx HTTP 状态、明确错误对象或无法识别的顶层响应均为失败。
- 第三方原始错误正文不得直接发送到前端或写入普通日志。

### 5.5 动态签名隔离

官方页面当前使用动态 CRC32 相关逻辑生成签名，该逻辑不等同于标准 CRC32。应用不得根据观察结果自行重写签名算法，而应在隔离的隐藏 WebView 中执行官方页面提供的脚本。

隐藏 WebView 必须满足：

- 只允许加载已校验的官方 HTTPS 来源。
- 禁止跨来源导航、弹窗、新窗口和下载。
- 不向页面注入 Tauri IPC 或任何应用能力。
- 仅由 Rust 主动执行受控脚本并读取有大小限制的返回值。
- 初始化、单次调用和返回值均有边界；初始化失败可重试，重试前销毁旧实例。
- 页面脚本变化导致无法签名时，返回稳定的初始化或请求错误，不回退到自实现签名。

隐藏窗口的内部消息封装、函数名和复用方式由实施计划决定。

### 5.6 网络与资源安全

- 搜索、URL、封面和歌词接口共享一个有界的全局限流器；首版初始额度见附录 B。
- 收到 429 时尊重 `Retry-After`；缺失时进入有上限的退避等待。等待过程必须可取消。
- GD API 只允许预定的 HTTPS 地址。
- 媒体地址只允许 HTTPS；HTTP 地址可以尝试升级为同主机 HTTPS，但不得回退为明文传输。
- 媒体请求必须拒绝环回、私有、链路本地、组播和未指定地址，并防止 DNS 重绑定；每次重定向都重新校验目标。
- 按资源类型设置适用的连接、空闲或总时限，并限制重定向次数、响应体、媒体文件和磁盘余量；具体见附录 B。
- 资源大小按实际解压后的字节计数，不能只信任 `Content-Length`。
- 封面只接受支持的位图格式，不接受 SVG；同时限制像素尺寸和字节大小。

首版初始数值见附录 B。

## 6. 领域状态、并发一致性与错误模型

### 6.1 搜索与目录状态

搜索至少包含空闲、加载、成功、空结果和失败五种可见状态。

- 每次搜索具有唯一请求身份，只有当前最新请求可以更新当前界面结果。
- 搜索条件变化后，旧结果立即失效；仍在执行的旧请求应取消或在完成时被丢弃。
- 每次成功搜索生成不可变 `SearchSnapshot`，记录归一化关键词、音源、模式、期望数量和搜索时间。
- 下载目录准备是独立状态：成功搜索或基础目录变化后进行扫描，经历空闲、扫描中、就绪或失败；扫描失败不能伪装成无重复文件。

### 6.2 下载任务与队列

加入队列时创建不可变下载任务，至少包含：

- 搜索快照与歌曲快照。
- 音源、音质和基础目录。
- 是否写入封面、是否下载歌词。
- 当前状态、尝试次数、错误信息与已下载字节。

加入队列后改变界面配置，不影响已经排队或正在执行的任务。

下载状态至少覆盖：等待、限流冷却、下载中、取消中、成功、失败、跳过和已取消。队列只有一个消费者，任何时刻最多一个音乐任务处于主动下载阶段。

- “取消当前”只取消当前任务，清理完成后继续下一项。
- “取消全部”取消当前任务，并把所有未开始任务转换为“已取消”终态；任务仍保留在本次统计集合中。界面等待后端完成必要清理后再显示队列已停止。
- 失败任务由用户手动重试；重试沿用原任务快照，不增加队列总任务数。
- 429 引起的限流等待可自动恢复；其他任务错误不得进行无边界自动重试。
- 音频文件完成最终提交即锁定任务成功结果。任务在可选歌词处理完成或被取消后才发布“成功”终态；封面或歌词失败、以及音频提交后的附件取消只产生警告，不能把任务改为失败或已取消。首版没有“部分成功”状态和附件修复流程。

活动状态包括限流冷却、下载中和取消中。统计必须始终满足：

`总数 = 等待数 + 活动数 + 成功数 + 失败数 + 跳过数 + 已取消数`

且活动数不得大于 1。

### 6.3 跨层状态一致性

基于第 4.1 节的权威状态边界，前端必须能够在事件丢失、乱序、窗口挂起或重新挂载后，请求并恢复一份完整、一致的状态快照。

增量事件必须可幂等处理，并能关联到明确的请求或任务。具体采用修订号、序列号、缓冲或其他机制，由实施计划决定，但不得只依赖“事件按顺序且永不丢失”的假设。

### 6.4 错误处理

| 场景 | 产品行为 |
| --- | --- |
| 启动关键初始化失败 | 阻止进入主界面，显示稳定原因与错误码，并提供重试、打开日志目录、退出 |
| 更新检查失败 | 按第 9 章显示非阻塞警告，不影响搜索与下载 |
| 搜索失败 | 保留输入条件，显示可操作错误，允许重试 |
| 单个下载失败 | 标记该任务失败，继续下一项，允许手动重试 |
| 封面或歌词失败 | 音频任务保持成功，显示附件警告 |
| 文件系统失败 | 显示目标路径上下文和可执行建议，不暴露无关内部信息 |
| 触发限流 | 显示冷却状态与预计可恢复时间，允许取消 |
| 签名运行环境失败 | 显示稳定初始化或请求错误，不静默更换算法 |

错误跨越 Rust 与前端边界时，Rust 只返回稳定错误码和经过清理的参数；前端从集中式中文文案资源生成用户消息。日志可记录诊断上下文，但不得记录签名材料、密钥、更新私钥、密码或未经清理的第三方响应。

首版至少提供以下稳定领域条件码；实现可以增加更细子码，但不能把这些用户行为重新合并为无差别错误：

| 条件码 | 分类 | 状态落点 |
| --- | --- | --- |
| `INPUT_INVALID` | 命令参数不符合白名单 | 当前命令拒绝，不启动领域操作 |
| `INIT_LOG` | 日志初始化失败 | 阻塞启动 |
| `INIT_SIGNATURE` | 签名运行环境初始化失败 | 阻塞启动 |
| `MUSIC_SIGNATURE` | 单次签名调用失败 | 当前搜索或音乐请求失败 |
| `MUSIC_HTTP` | 音乐接口网络或 HTTP 失败 | 当前搜索或音乐请求失败 |
| `MUSIC_RATE_LIMITED` | 音乐接口限流 | 非终态限流冷却，可取消并可恢复 |
| `MUSIC_SCHEMA` | 音乐接口响应格式不兼容 | 当前搜索或音乐请求失败 |
| `MUSIC_QUALITY_UNAVAILABLE` | 所选音质不可用或明确降级 | 当前下载任务失败 |
| `DOWNLOAD_NETWORK` | 媒体传输失败 | 当前下载任务失败 |
| `DOWNLOAD_MEDIA` | 媒体格式识别或处理失败 | 当前下载任务失败 |
| `FS_PATH` | 目录、路径、空间或权限失败 | 当前命令或任务失败 |
| `FS_COMMIT_CONFLICT` | 最终文件在提交前出现 | 当前下载任务终态为跳过 |
| `FS_COMMIT_UNSUPPORTED` | 文件系统不支持安全无覆盖提交 | 当前下载任务失败 |
| `UPDATE_CHECK` | 更新检查或清单失败 | 非阻塞更新警告 |
| `UPDATE_INSTALL` | 更新下载、验证或安装失败 | 更新失败，当前版本继续运行或安全退出 |

### 6.5 退出与更新安装屏障

普通关闭和更新安装共用退出屏障。只要存在未终止的音乐任务或正在下载的更新包，应用都不得直接退出或开始安装。

用户可以：

- 返回应用继续任务；或
- 确认取消并退出，等待后端完成可安全完成的取消和临时文件清理。

如果平台更新器无法可靠取消正在进行的更新下载，应等待其进入安全状态，而不是强行退出。更新安装只能在屏障通过后启动。

进程崩溃或系统强制终止无法保证清理；这类情况下，由下次扫描按第 7.4 节规则识别并清理应用自己的陈旧临时文件。

## 7. 下载存储、媒体处理与日志

### 7.1 下载目录

默认基础目录：

- Windows：系统“音乐”目录。
- macOS：`~/Music`。

实际输出目录为：

`<基础目录>/<清理后的搜索关键词>/`

关键词、歌曲名称和歌曲 ID 都是不可信路径输入，必须转换为确定、安全的文件名片段，并处理跨平台非法字符、路径穿越、保留名称、尾随点或空格、空结果、长度限制和清理后碰撞。具体字符转换算法在实施计划中确定并以测试固定。

### 7.2 文件命名与实际格式

音频文件名：

`【安全表示的歌曲 ID】清理后的歌曲名称.<实际音频扩展名>`

歌词文件使用相同主文件名和 `.lrc` 扩展名。安全表示只改变文件系统表达，不得改变第 7.3 节去重键中的歌曲身份。

实际扩展名必须根据下载内容和支持的媒体格式可靠识别，不得假定所有音频均为 MP3，也不得为了匹配扩展名进行转码。

### 7.3 去重

去重键固定为：

`歌曲 ID + 清理后的歌曲名称 + 实际音频扩展名`

去重规则：

- 只把已经存在的最终音频文件视为完成项；临时文件和歌词文件不参与去重。
- 同一 ID 与名称但实际扩展名不同的文件可以并存。
- 最终目标已存在时跳过，不覆盖也不重命名为副本。
- 去重上下文由归一化基础目录和清理后的关键词目录共同确定。
- 搜索成功或基础目录改变后重新扫描；成功提交后立即更新后端去重状态。
- 即使扫描时不存在，最终提交仍必须防御其他进程或并发任务刚刚创建同名文件的情况。

### 7.4 临时文件与无覆盖提交

- 在目标目录内以唯一名称创建 `.yinmi-<UUID>.part` 临时文件，并使用“仅当不存在时创建”语义。
- 下载、格式识别、元数据和封面写入均在临时文件上完成。
- 最终提交前刷新并关闭临时文件。
- 使用操作系统支持的原子、无覆盖提交语义将临时文件变为最终文件。
- 如果最终目标在提交前已出现，删除自己的临时文件并把任务记为“跳过”。
- 如果目标文件系统不能提供所需安全语义，任务应明确失败，不得退化为先检查再覆盖式重命名。
- 取消与最终提交必须具有明确线性化结果：一旦最终文件安全提交成功，任务为成功；否则取消必须清理自己的临时文件。
- 正常失败与取消只清理当前任务创建的临时文件。
- 目录扫描时只清理名称匹配应用格式、且超过陈旧阈值的临时文件，不删除无法证明归属的文件。

陈旧阈值见附录 B。

### 7.5 封面、歌词与媒体元数据

- 封面直接嵌入音频文件，不额外保存图片文件。
- MP3 使用 ID3 图片帧，FLAC 使用 Picture Block；其他格式只有在媒体库明确支持安全写入时才启用封面。
- 歌词作为独立 `.lrc` 文件保存，内容来自原文 `lyric` 字段。
- 只在音频最终提交成功后生成歌词，避免音频失败时留下孤立歌词。
- 封面或歌词获取、验证、写入失败时记录附件警告，不回滚已经安全提交的音频，也不把音频任务改为失败。
- 任何音频元数据改写都在临时文件阶段完成，避免破坏已存在的最终文件。
- 歌词同样先写入目标目录内的应用临时文件，再使用无覆盖提交；已有同名 `.lrc` 时保留用户文件、删除自己的临时文件并记录附件警告。

### 7.6 日志

日志目录：

- Windows：`%LOCALAPPDATA%/音觅/log/`
- macOS：`~/Library/Logs/音觅/`

应用使用滚动日志；开发运行、手动构建和安装后的正式程序都遵守相同的平台目录规则，不写到可执行文件旁。日志至少覆盖初始化、音源、请求阶段、限流、歌曲 ID、下载阶段、媒体处理、更新检查和错误链。界面行为见第 8.7 节，初始内存上限见附录 B。

## 8. 用户界面与交互

### 8.1 启动页

启动页依次呈现必要初始化过程，包括日志、签名运行环境、音源、下载引擎和更新检查状态。更新检查不得阻塞进入主界面。

为避免正常快速启动时闪烁，只有启动超过短暂延迟后才显示详细状态。关键初始化失败时，启动页必须展示稳定错误原因和错误码，并提供重试、打开日志目录和退出操作。

### 8.2 搜索与下载配置

| 控件 | 默认值与约束 |
| --- | --- |
| 搜索关键词 | 空；搜索时必填 |
| 音源 | 网易云音乐 |
| 搜索模式 | 单曲匹配 |
| 搜索数量 | 20；整数，范围 1–1000 |
| 下载音质 | 320 kbps |
| 写入封面 | 开启 |
| 下载歌词 | 开启 |
| 基础下载目录 | 当前系统音乐目录 |

三种搜索模式旁显示固定说明：“三种模式均返回歌曲列表，不提供专辑或歌单容器浏览”，避免用户误解模式结果。

搜索数量表示希望最终获得的歌曲数。后端可使用分页实现；结果表格必须能在最多 1000 条结果下保持滚动、选择、全选和状态更新流畅，具体采用虚拟滚动或其他机制由实施计划决定。首版初始性能预算见附录 B，并在第 12 章第一阶段固定基准环境。

界面只显示基础下载目录。修改搜索条件会使当前搜索结果失效；修改音质、封面、歌词等下载选项不会改变搜索结果。

队列存在未终止任务时，锁定关键词、音源、模式、搜索数量和新搜索操作；用户必须先取消全部并等待队列停止，才能更换搜索上下文。音质、封面、歌词和基础目录仍可调整，但必须明确标注“仅用于下一批加入的任务”，且绝不影响已入队任务。

首版不持久化这些配置，重启后恢复默认值。

### 8.3 搜索结果

结果表格至少显示：选择框、序号、歌曲名、艺人、专辑、音源、时长、可用能力提示和当前下载状态。歌曲 ID 不单独显示，但保留在领域模型中。缺失时长显示 `--:--`。界面同时显示结果总数和已选数量，支持逐项选择、全选当前结果、取消全选和将所选歌曲加入下载。

没有有效结果时显示明确空状态；部分记录因格式异常被跳过时显示汇总警告，但不向用户倾倒原始响应。

### 8.4 下载区

固定下载栏至少显示：

- 总任务数、等待数、当前活动任务、成功数、失败数、跳过数、取消数。
- 当前歌曲、状态、已下载字节、总字节（若已知）、速度和进度。
- 取消当前、取消全部、重试单个失败、重试全部失败和打开下载目录。

当总字节未知时，使用不确定进度状态，不能伪造百分比。限流冷却、取消中和附件警告必须有明确文本，不能只靠颜色区分。

### 8.5 视觉与布局

视觉基调为轻量、清晰、偏浅蓝的桌面工具：

| 用途 | 颜色 |
| --- | --- |
| 页面背景 | `#F6F9FC` |
| 卡片背景 | `#FFFFFF` |
| 搜索区域背景 | `#E7F0F8` |
| 主按钮背景 | `#0B78D0` |
| 主按钮文字 | `#FFFFFF` |
| 链接与强调文字 | `#075FA8` |
| 固定下载栏 | `#16283E` |
| 成功状态 | `#13785C` |

中文界面优先使用系统无衬线字体栈，数字与进度信息使用易读的等宽数字特性；日志使用平台等宽字体栈。首版仅提供浅色主题。

主界面从上到下为标题栏、搜索配置区、结果表格和固定下载栏；日志以抽屉或模态区域呈现。目标窗口尺寸为 1280 × 800，最小尺寸为 800 × 480。宽度低于 1040 时搜索配置改为多行排列；结果表格可水平滚动；下载栏内容超过可用高度时内部滚动，且不超过窗口高度的 40%。

### 8.6 可访问性

- 所有可交互元素都有可见键盘焦点。
- 普通文本对比度至少 4.5:1，非文本控件与焦点指示至少 3:1。
- 状态不得只用颜色表达。
- 错误、队列状态和进度使用合适的 `alert`、live region 与 progress 语义，并限制高频进度播报。
- 图标按钮提供可读名称；未知总量的进度明确标为不确定状态。
- 模态日志抽屉或确认对话框具备对话框语义、初始焦点、焦点约束、背景不可交互、Escape 关闭规则和关闭后焦点恢复。
- 尊重系统“减少动态效果”设置。

### 8.7 模态日志抽屉

主界面提供“运行日志”入口，打开右侧模态日志抽屉，不跳转页面。抽屉支持：

- 当前运行日志实时追加。
- 按文本、歌曲 ID 或模块搜索。
- 按全部、信息、警告和错误级别筛选。
- 自动滚动开关、复制全部和打开系统日志目录。

界面只保留附录 B 规定数量的近期日志，完整滚动记录以磁盘文件为准。筛选和搜索不修改日志文件。关闭抽屉后恢复原布局与触发按钮焦点。

### 8.8 应用图标

正式图标采用已确认的“寻声环”方向，不使用常见音乐音符。图形由信号蓝圆角底板、同心扫描轨迹、中心声源点和右上方绿色命中点组成，表达搜索、声波与命中。

图标源文件使用 SVG 保存，并生成 Windows ICO、macOS ICNS 和各级 PNG。16–32 px 版本简化为外环、中心点和命中点，移除内层细节。图标色彩沿用界面令牌：信号蓝 `#0B78D0`、冷白 `#F7FBFF`、浅声波蓝 `#9BDEFF` 和命中绿 `#64D2AF`。

## 9. 应用内更新与版本

- `tauri.conf.json` 中的语义化版本号是应用、安装包和更新清单的单一版本来源。
- 首个公开测试版本为 `0.1.0`；发布标签使用对应的 `v0.1.0`。
- 启动后在后台检查并下载可用更新，不阻塞搜索与音乐下载；更新包验证完成后允许“立即重启安装”或“稍后”。
- “稍后”表示本次运行不再强制提示，可在退出时安装，或在下一次启动重新提示。
- 更新检查或下载失败不阻塞正常功能，并提供可重试提示。
- 不允许降级安装。
- 更新端点固定为 `https://github.com/xiaobing6/yinmi-desktop/releases/latest/download/latest.json`。
- 仅用于首发更新演练的 `0.0.1` 测试构建可以通过隔离配置指向测试清单；任何生产构建都必须恢复上述固定端点，发布工作流必须检测并拒绝测试端点泄漏。
- Windows 更新目标键为 `windows-x86_64`。
- macOS 更新目标必须明确使用 `macos-universal`，Intel 与 Apple Silicon 运行同一 Universal 更新包。
- 首个发布版本即启用 Tauri 更新签名验证。
- 公钥嵌入应用，私钥和密码仅保存在 GitHub Actions Secrets 中。
- 下载完成后的安装必须通过第 6.5 节退出屏障。

## 10. 构建、发布与平台签名

### 10.1 发布产物

| 平台 | 安装产物 | 更新产物 |
| --- | --- | --- |
| Windows x64 | NSIS `.exe` | 对应更新包与 `.sig` |
| macOS Universal | `.dmg` | Universal `.app.tar.gz` 与 `.sig` |

macOS 使用 `universal-apple-darwin` 目标构建，不能分别发布 Intel 与 Apple Silicon 更新包。

`latest.json` 至少包含版本、发布日期、更新说明，以及 `windows-x86_64` 和 `macos-universal` 两个平台条目的下载地址与签名。若发布工具默认生成分架构 Darwin 键，工作流必须生成并验证符合本设计的清单。

### 10.2 发布流程

1. 发布标签、配置版本和产物版本完全一致。
2. Windows 与 macOS 从同一提交构建。
3. 工作流验证所有安装包、更新包、签名和 `latest.json` 的完整性及平台键。
4. 先创建 Draft Release，人工检查后发布为普通正式 Release，不标记为预发布。
5. 应用更新端点始终指向最新正式 Release。
6. 已发布版本不覆盖同名资产；修复发布问题必须提升版本号后重新发布。

### 10.3 平台签名说明

- 首版 Windows 安装包不使用 Authenticode，安装时可能出现 SmartScreen 提示；发布说明需给出来源校验指引。
- 首版 macOS 使用 ad-hoc 签名，不做 Developer ID 公证，首次打开可能需要用户在系统设置中允许。
- Tauri 更新签名独立于操作系统代码签名，首版仍必须启用。
- 后续引入商业代码签名或 Apple Developer ID、公证时，不改变本设计中的更新签名和产物一致性要求。

## 11. 验证与验收

### 11.1 自动化测试

前端持续执行：

- Prettier、ESLint、`svelte-check`、TypeScript 检查和 Vite 构建。
- Vitest 与 Testing Library 测试默认值、数量范围、三种搜索模式、最新请求生效、目录状态、选择逻辑、队列统计、状态重同步、关键可访问性语义和设计颜色。

Rust 持续执行：

- `cargo fmt --check`、Clippy 和单元测试。
- 使用 Wiremock 固定请求正文与响应归一化契约。
- 测试网络边界、路径清理、去重、无覆盖提交、队列状态、取消、限流、退出屏障以及附件失败不改变音频成功语义。
- 在实际临时文件系统中测试目标预先存在和提交前竞争出现两种场景，确保永不覆盖。

### 11.2 持续集成

- 功能分支推送执行快速静态检查与单元测试。
- Pull Request 和 `master` 执行完整矩阵：Windows x64 运行 Windows 目标与文件系统测试；macOS 同时检查 Intel、Apple Silicon 目标，并运行 macOS 文件系统测试。
- 普通 PR 不要求发布签名，也不上传可发布安装资产。
- 受保护分支使用稳定的必需检查名称；若仓库启用 merge queue，同一工作流支持 `merge_group`。
- Release 工作流从干净环境重新验证后构建正式产物，不能复用未经发布验证的本地文件。

## 12. 实施计划入口

最终实施计划应按可验证的纵向成果拆分，避免把全部工作压在一个大阶段中：

1. 可行性验证、基础 CI 与工程脚手架。
2. 共享契约、安全基础设施与状态快照。
3. 搜索纵向链路：配置、签名、接口、归一化、状态同步和结果界面。
4. 下载纵向链路：任务、队列、媒体、存储、状态同步和下载界面。
5. 集成打磨、日志、可访问性、生命周期与应用内更新。
6. 跨平台发布、持续集成和最终验收。

在大规模实现前，第一阶段必须用最小原型验证以下高风险假设：

- GD 音乐台对期望数量 1000 的实际支持方式，包括单次限制、分页、重复和终止条件。
- 隐藏 WebView 在 Windows 与 macOS 上加载官方页面、执行受控签名调用和彻底隔离 Tauri 能力的可行性。
- Reqwest/Rust 网络栈中 DNS 解析、重定向与地址复核能否满足防 DNS 重绑定要求。
- Windows 与 macOS 目标文件系统上的原子无覆盖提交方案。
- Tauri 更新下载的取消能力，以及与统一退出屏障协作的安全状态。

第一阶段是硬性退出门，至少产出：已核验的协议夹具与请求顺序、分页与安全页数决策、首版媒体容器允许列表、1000 条结果的基准环境与测量结果、五项高风险原型结论、基础 Windows/macOS 烟雾构建，以及必要的设计决策记录。更新原型必须明确证明“可取消且清理完成”，或给出“不可取消时的最大安全等待、超时/失败落点和用户反馈”；无法形成有界行为时必须回修设计。任一结论不满足本文要求时，先修订并重新审核设计，不得直接进入第二阶段。

设计最终确认后，先只编写第一阶段的详细实施计划和总体里程碑路线图。第二至第六阶段的详细计划在第一阶段门控通过后分别编写，避免基于未经验证的文件路径、API 或平台机制提前锁定实现。

可行性验证可以改变内部机制和实施拆分，但如果需要改变本文规定的用户行为、安全不变量、外部契约或发布结果，必须先修订设计文档并重新审核。

### A.2 表单编码规则

关键词或 ID 先使用 JavaScript `encodeURIComponent` 等价规则编码，再对 `!'()*` 做百分号编码。编码后的值同时用于原始表单字段和签名输入，不得二次编码。

实现必须使用专用正文构造器写入已经编码的值，不能再交给会自动百分号编码的普通表单序列化器。

以下请求正文按 2026-07-14 官方页面实际脚本的字段顺序固定，并用于契约测试。`<signature>` 分别是编码后的关键词或 ID 经官方 `crc32()` 生成的返回值：

```text
types=search&count=20&source=netease&pages=1&name=%E5%91%A8%E6%9D%B0%E4%BC%A6&s=<signature>
```

```text
types=search_album&count=20&source=netease&pages=1&name=%E5%91%A8%E6%9D%B0%E4%BC%A6&s=<signature>
```

```text
types=search_playlist&count=20&source=netease&pages=1&name=%E5%91%A8%E6%9D%B0%E4%BC%A6&s=<signature>
```

```text
types=url&id=123456&source=netease&br=320&s=<signature>
```

```text
types=pic&id=123456&source=netease&size=300&s=<signature>
```

```text
types=lyric&id=123456&source=netease&s=<signature>
```

特殊字符编码也必须作为逐字节黄金样例固定：

| 原始值 | 编码后值 |
| --- | --- |
| `A B!'()*/?=%` | `A%20B%21%27%28%29%2A%2F%3F%3D%25` |
| `id/42?x=1%` | `id%2F42%3Fx%3D1%25` |

编码后的 `%` 必须原样进入请求正文与签名输入。例如第一个值用于搜索时，`name` 字段必须包含 `A%20B%21%27%28%29%2A%2F%3F%3D%25`，不能被再次编码为以 `%25` 开头的序列。
